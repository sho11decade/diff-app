# 実装計画

## 1. 計画の前提
- 対象: 間違い探し生成APIの研究基盤化
- 現状: [main.py](main.py) にAPIと生成ロジックが集約
- 制約: 4GB VRAM環境を主軸に設計
- 目標: 研究再現性（seed固定）、説明可能性（trace）、比較可能性（モジュール分離）を備えた実装に移行

## 2. マイルストーン

### M1: パイプライン分離（Week 1-2）
成果:
- API層と生成ロジック層を分離
- 既存 `/generate` の機能互換を維持

完了条件:
- エンドポイントの入力/出力仕様を維持
- 単体テストで主要関数が独立実行可能
- 回帰確認（同一入力で既存仕様のレスポンス形式が一致）

### M2: Trace対応と再現性（Week 3）
成果:
- `seed` 指定対応
- `trace=true` でスコア内訳と編集根拠を返却
- 実験ログをファイル保存

完了条件:
- 同一画像 + 同一seedで位置情報が一致
- traceログに編集タイプ、領域特徴、実行時間が記録
- ログ保存失敗時のフォールバック（API本体は失敗させない）

### M3: 難易度スコアリング実装（Week 4-5）
成果:
- ルールベース難易度スコア関数導入
- easy/medium/hard の編集ポリシー切替

完了条件:
- 難易度ごとに選択領域と編集種別の分布が変わる
- スコア内訳がレスポンスとログに出力
- 既存性能要件（1枚30秒以内）を満たす

### M4: 研究評価基盤（Week 6-7）
成果:
- オフライン評価スクリプト（LPIPS/SSIM/処理時間）
- アブレーション実験設定（サリエンシー有無、編集器構成）

完了条件:
- 評価レポートをCSV/Markdownで出力
- 最低3条件の比較結果を再生成可能

### M5: 高度編集（任意拡張、Week 8+）
成果:
- inpainting編集器（初期はCPU fallback許容）
- 高負荷処理の非同期化検討

完了条件:
- 低負荷編集に影響なく機能フラグで切替可能
- 失敗時に既存編集器へフォールバック

## 5. リスクと緩和策
- VRAM不足: 軽量モデル優先、解像度制限、タイル推論
- 生成品質の不安定性: 編集器を複数保持しフォールバック
- 再現性崩れ: 乱数源を一元化、バージョン固定
- 実験ログ肥大化: 日次ディレクトリ分割、保持期間設定

## 6. Definition of Done
- API互換を維持しながらモジュール分離済み
- `seed` と `trace` が動作し、ログが保存される
- テスト（単体/統合/再現性）が通過
- 研究比較用の評価出力（CSVまたはMarkdown）が生成できる
