# 研究価値を創出するシステム全体アーキテクチャ提案

## 1. 目的と設計方針
本提案は、要件定義で示された「説明可能な間違い探し生成」を、研究として再現可能・評価可能な形で実装するための全体アーキテクチャである。

重視する設計方針:
- 説明可能性: どの領域を、なぜ、どの編集で変更したかをログとして残す。
- 再現性: 同一入力・同一シードで同一出力を再生成できる。
- 比較可能性: 手法差分をA/B比較できるモジュール分離を行う。
- 省VRAM運用: 4GB VRAM環境でも段階的に実験可能な構成にする。
- 研究拡張性: 新モデル追加をプラグイン的に行える。

## 2. 全体像（レイヤ化）
システムを6レイヤで構成する。

1. API/Orchestration Layer
- FastAPIで /generate を提供。
- リクエスト検証、認証、レート制限、実験ID発行を担当。

2. Data & Preprocessing Layer
- 画像デコード、サイズ制約適用、正規化、キャッシュキー生成。
- 低解像度推論と高解像度編集の二段処理をサポート。

3. Scene Understanding Layer
- セマンティックセグメンテーション（DeepLabv3+ / SegFormer）
- サリエンシー推定（U2Net / BASNet）
- 領域特徴抽出（面積、エッジ密度、色分散、クラスラベル）

4. Decision Layer
- 候補領域のスコアリング（難易度スコア関数）
- 編集手法の選択ポリシー（色変化、幾何変換、inpainting）
- 安全性制約（顔・文字・高重要領域の改変回避ルール）

5. Editing & Compositing Layer
- ROI編集の実行。
- 境界整合（Poisson/seamless clone相当）
- 位置情報と回答画像生成。

6. Evaluation & Research Layer
- 自動評価（LPIPS/SSIM/領域可視性指標）
- ユーザ評価ログ（検出時間、正答率）
- 実験トラッキング（設定・重み・乱数シード・推論時間）

## 3. 研究価値を生む中核設計

### 3.1 Difficulty-Controllability by Design
難易度を後付けではなく設計時点で定義する。候補領域 r に対して:

Difficulty(r) = w1 * (1 - Saliency(r)) + w2 * SemanticImportance(r) + w3 * EditComplexity(r) + w4 * ContextCrowdedness(r)

- easy: 高サリエンシー領域 + 低複雑編集
- medium: 中サリエンシー + 幾何変換中心
- hard: 低サリエンシー領域 + inpainting/複合編集

研究貢献候補:
- 難易度ラベルと人間検出時間の相関を検証し、スコア関数を学習的に較正する。

### 3.2 Explainable Difference Card
各差分に「差分カード」を付与:
- 対象クラス（例: chair）
- 編集タイプ（color/affine/inpaint）
- スコア内訳（saliency寄与、complexity寄与）
- 生成根拠画像（マスク、サリエンシーマップ）

これにより説明可能AIとして論文化しやすい成果物になる。

### 3.3 Hybrid Editor Policy
編集器を1つに固定せず、ポリシーで選択する:
- Tier 1: 低計算（色・明度・微小移動）
- Tier 2: 中計算（アフィン、局所ワープ）
- Tier 3: 高計算（inpainting）

研究貢献候補:
- 同難易度でも編集器の違いが検出率に与える影響を比較できる。

### 3.4 Resource-Aware Inference
4GB VRAMに対応するため:
- セグメンテーションは軽量バックボーン優先。
- mixed precision、tile推論、バッチサイズ1固定。
- 高負荷編集はCPU fallbackまたは非同期キューで実行。

研究貢献候補:
- 低リソース環境での品質-速度トレードオフ曲線を提示可能。

## 4. コンポーネント設計

### 4.1 推奨ディレクトリ分割
- app/api: FastAPIルータ、リクエスト・レスポンススキーマ
- app/core: 設定、認証、レート制御、乱数管理
- app/pipeline: 前処理、理解、意思決定、編集、後処理
- app/models: セグメンテーション/サリエンシー推論ラッパ
- app/research: 評価指標、実験記録、可視化
- tests: 単体・統合・性能・再現性テスト

### 4.2 Pipelineインタフェース（研究拡張向け）
- Segmenter: predict(image) -> mask, class_map
- SaliencyEstimator: predict(image) -> saliency_map
- RegionScorer: score(regions, maps, difficulty) -> ranked_regions
- Editor: apply(image, region, policy) -> edited_image, metadata
- Evaluator: evaluate(src, out, meta) -> metrics

この抽象化で、モデル差し替え時にAPI層を変更せず実験できる。

## 5. API仕様の研究拡張
既存 /generate に加えて、研究用途で以下を提案:
- POST /generate?trace=true
  - 返却に trace_id, difference_cards, score_breakdown を追加。
- GET /experiments/{trace_id}
  - 設定、中間出力、評価指標を取得可能。

これにより「生成結果」だけでなく「生成理由」を取得できる。

## 6. 評価設計（研究プロトコル）

### 6.1 オフライン評価
- 品質: LPIPS, SSIM
- 局所自然性: 編集境界付近の知覚差分指標
- 性能: end-to-end latency、GPU/CPU使用率

### 6.2 人間評価
- 被験者タスク: 間違い検出時間と正答率
- 難易度妥当性: easy/medium/hard の統計的分離
- 説明可能性: 差分カードの納得感アンケート

### 6.3 アブレーション
- サリエンシー無し vs 有り
- 単一編集器 vs ハイブリッド編集器
- 固定閾値選択 vs スコア選択

## 7. 実装フェーズ

### Phase 1: ベースライン（2-3週間）
- 現行FastAPIにパイプライン層を導入
- 低計算編集器（color/affine）
- traceログ保存（JSON）

### Phase 2: 研究強化（3-5週間）
- サリエンシーモデル統合
- 難易度スコア関数導入
- difference card生成

### Phase 3: 高度化（4-6週間）
- inpainting導入
- 実験管理とアブレーション自動化
- 人間評価UI/集計スクリプト

## 8. 成果物定義（論文化を意識）
- 再現可能コード（seed固定、実験設定保存）
- データセット分割と評価プロトコル
- 比較実験表（難易度制御、自然性、速度）
- 失敗事例分析（どの条件で不自然化するか）

## 9. 本リポジトリ向け即時アクション
1. 現行 main.py のロジックを pipeline モジュールへ分割。
2. 生成時に差分メタデータ（編集種別、スコア、領域特徴）を返す。
3. 実験ログを experiments/YYYYMMDD/trace_id.json に保存。
4. 再現性のため APIで seed 指定を受け付ける。

この4点で、単なるデモAPIから「研究可能なシステム基盤」へ移行できる。
